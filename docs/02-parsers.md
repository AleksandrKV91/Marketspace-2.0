# Парсеры Excel → Supabase

## Общие правила для всех парсеров

```typescript
// ВСЕГДА
XLSX.read(buffer, { cellDates: false, raw: true })

// Дата из Excel serial → ISO
function excelToISO(serial: number): string {
  if (!serial || typeof serial !== 'number') return ''
  const date = new Date((serial - 25569) * 86400 * 1000)
  return date.toISOString().split('T')[0]
}

// Нормализация заголовка (убирает пробелы, нижний регистр)
function norm(s: unknown): string {
  return String(s ?? '').trim().toLowerCase()
}

// Безопасное число
function toNum(v: unknown): number | null {
  const n = Number(v)
  return isNaN(n) || v === null || v === '' ? null : n
}
```

## 2. Отчёт по SKU (`parseSkuReport`)

**Файл**: `Отчет_по_SKU_*.xlsb`, лист `Лист7`
**Периодичность**: раз в 3-5 дней

### Структура листа

```
Строка 0: группы (Планирование / Выручка Total / ДРР Total / ...)
Строка 1: заголовки (Затраты план / 46105 / 46104 / ... / Остаток на ВБ ФБО / ...)
Строка 2+: данные SKU
```

### Что берём (по Основным показателям, лист "отчет по ску")

**Характеристики** (берём из отчёта, т.к. свежее Свода):
- Дата появления на полке
- Менеджер
- Статус Новинки

**Игнорируем** (берём из Свода): Категория, Предмет, Название, Бренд, Менеджер НГ, Сезон

**Дневные метрики** (5 дней, колонки = Excel serial dates):
- Затраты за 5 дней (ad_spend)
- Выручка Total за 5 дней (revenue)
- ДРР Total за 5 дней (drr_total)
- ДРР Рекламный за 5 дней (drr_ad)
- CTR за 5 дней (ctr)
- CR в корзину за 5 дней (cr_cart)
- CR в заказ за 5 дней (cr_order)
- CPM за 5 дней (cpm)
- CPC за 5 дней (cpc)
- Доля рекламных заказов (ad_order_share)
- СПП (цена после скидки, одна на дату)

**Снапшот** (на дату последней загрузки, "используем если последняя дата актуальная"):
- Остаток на ВБ ФБО
- Остаток FBS Пушкино
- Остаток FBS Смоленск
- остаток комплектов
- остаток, дни
- дней до прихода
- Запас дней до OOS
- Маржа Опер.
- ЧМД за пять дней, руб
- Затраты план
- ДРР план
- Поставка план. (дата)
- Поступ-ния шт. план
- Цена

**НЕ берём**: Маржа %, Выручка план, Изменение цены (колонки 87-92)

### Определение смещения колонок `pos`

```typescript
// Заголовок "Затраты план" должен быть в col 32 (0-indexed)
// Если структура файла изменилась — он сместится
const pos = (() => {
  for (let ci = 27; ci < 45; ci++) {
    if (norm(headerRow[ci]).includes('затраты план') ||
        norm(headerRow[ci]).includes('spend plan')) {
      return ci - 32
    }
  }
  // fallback: искать в groupRow
  for (let ci = 27; ci < 45; ci++) {
    if (norm(groupRow[ci]).includes('планирован')) {
      return ci - 32
    }
  }
  return 0
})()

// Колонка последней даты (данные актуальны)
const POS_LATEST = pos < 0 ? 26 : 29
```

## 3. Таблица остатков (`parseStock`)

**Файл**: `Таблица_Остатков_*.xlsb`, лист `sheet1`
**Периодичность**: хоть каждый день

### Структура листа

```
Строки 0-4: пустые
Строка 5:   заголовки
Строка 6+:  данные SKU
```

### Что берём (по Основным показателям, лист "Остатки")

**Из первых колонок:**
- SKU (col 2) → sku_wb
- дата появления на полке (col 9) → shelf_date (Excel serial)
- кол-во в поставке → supply_qty
- дата прихода → supply_date (строка DD.MM.YYYY)
- цена утром → price (последняя дата в файле)
- маржа → margin_pct

**Изменения цен** (колонки с датами в начале файла, после col 16):
- Берём только датированные колонки с изменениями цен (около cols 16-26)
- Берём только последнее вхождение каждой даты

**Остатки** (по названию заголовка, НЕ по индексу):
- `Остаток на ВБ ФБО` → fbo_wb
- `Остаток FBS Пушкино` → fbs_pushkino
- `Остаток FBS Смоленск` → fbs_smolensk

**Средние заказы и прогноз** (по названию заголовка):
- `средние заказы` → avg_orders (dpd)
- `прогноз заказов` → forecast

**Продажи по дням** (колонки LK-ABE):
- Загружаются **один раз** при первой загрузке файла
- При повторной загрузке — только новые даты (UPSERT по `(sku_wb, sale_date)`)
- Колонки BB-LI (`не используем`) — пропускаем
- Excel serial date → ISO

```typescript
// Определить диапазон LK-ABE по position в headerRow
// LK ≈ col 350, ABE ≈ col 730 (приблизительно)
// Точнее: искать первую дату > 44000 после col 340

// Брать все последовательные числовые даты начиная с firstSalesCol
```

## Общая логика API route (загрузка)

```typescript
export async function POST(req: NextRequest) {
  // 1. Auth check
  // 2. Read file
  // 3. Parse → validate → получить массив объектов
  // 4. Insert в uploads → получить upload_id
  // 5. UPSERT данных батчами по 500
  // 6. При ошибке → удалить upload запись (CASCADE удалит связанные данные)
  // 7. Вернуть { ok, upload_id, rows_count, diag: firstRow }
}
```
