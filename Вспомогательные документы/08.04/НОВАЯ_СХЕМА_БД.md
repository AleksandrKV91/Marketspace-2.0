## Текущее состояние (факт)

```
dim_sku             2812 строк   PRIMARY KEY = sku_ms (строка)
fact_sku_daily    176256 строк   UNIQUE KEY = (sku_ms, metric_date)
fact_sku_snapshot  49585 строк   UNIQUE KEY = (sku_ms, upload_id)
fact_stock_snapshot 2446 строк   UNIQUE KEY = (sku_wb, upload_id)
fact_abc            2586 строк   UNIQUE KEY = (sku_ms, upload_id)
fact_china_supply    500 строк   UNIQUE KEY = (sku_ms, upload_id)
uploads               22 строки
```

**Проблемы текущей структуры:**
1. `dim_sku` PRIMARY KEY = `sku_ms` — но основной идентификатор должен быть `sku_wb`
2. В `dim_sku` есть 2812 строк но `fact_sku_snapshot` 49585 — это 17 загрузок × 2763 ≈ 46971 строк + старые. Многовато.
3. Колонка `sku_ms` дублируется везде, но в ABC и Китай — только sku_ms, нет sku_wb

---

## Новая архитектура — принципы

### Главный ключ = sku_wb (артикул WB)

```
sku_wb = 174502899    ← число, уникальный ID товара на WB, НИКОГДА не меняется
sku_ms = "NWTPS100N3" ← внутренний артикул МС, нужен для JOIN с Отчётом по SKU и ABC
```

**Почему sku_wb главный:**
- Это официальный ID на WB — не меняется при переименовании SKU
- Таблица остатков работает только по sku_wb
- Свод содержит оба — это мост

**Почему нельзя полностью убрать sku_ms:**
- Отчёт по SKU содержит только sku_ms в col 0
- ABC анализ содержит только sku_ms
- Потребность Китай — только sku_ms
- Нельзя читать эти файлы без sku_ms

### dim_sku — единственный мост

```
dim_sku: sku_wb ↔ sku_ms (оба поля = оба ключа)
Заполняется из: Свод (единственный файл с обоими артикулами)
```

## Новая схема таблиц

### 1. dim_sku — справочник SKU

**Источник:** Свод (загружается первым, обязательно)

```sql
CREATE TABLE dim_sku (
  sku_wb        BIGINT PRIMARY KEY,           -- Артикул WB, главный ключ
  sku_ms        TEXT UNIQUE NOT NULL,         -- Артикул МС (из Отчёта/ABC/Китай)
  name          TEXT,                         -- Номенклатура
  brand         TEXT,                         -- Бренд
  subject_wb    TEXT,                         -- Предмет WB
  category_wb   TEXT,                         -- Категория WB
  supplier      TEXT,                         -- Поставщик
  country       TEXT,                         -- Страна производства
  nds_pct       NUMERIC,                      -- НДС %
  -- Сезонность по месяцам (коэффициенты 0-1)
  month_jan NUMERIC, month_feb NUMERIC, month_mar NUMERIC, month_apr NUMERIC,
  month_may NUMERIC, month_jun NUMERIC, month_jul NUMERIC, month_aug NUMERIC,
  month_sep NUMERIC, month_oct NUMERIC, month_nov NUMERIC, month_dec NUMERIC
);
-- Индекс для поиска по sku_ms (JOIN с fact_sku_daily)
CREATE INDEX idx_dim_sku_ms ON dim_sku(sku_ms);
```

**Что убрали:** `sku_warehouse` (= sku_ms, дублирование), `sku_china` (не используется в дашборде), `market_share`, `niche_appeal`, `availability`, `buyout_pct`, `avg_rating`, `seasonality`, `season_start`, `season_length`, `top_month`, `top_phrase`

### 2. fact_sku_daily — ежедневные метрики SKU

**Источник:** Отчёт по SKU (5 дат на загрузку)  
**Ключ:** (sku_ms, metric_date) — при пересечении дат, более поздняя загрузка перезаписывает

```sql
CREATE TABLE fact_sku_daily (
  id            BIGSERIAL,
  sku_ms        TEXT NOT NULL REFERENCES dim_sku(sku_ms),
  metric_date   DATE NOT NULL,
  -- Финансы
  revenue       NUMERIC,    -- Выручка Total за день, ₽
  ad_spend      NUMERIC,    -- Затраты на рекламу за день, ₽
  -- Воронка
  drr_total     NUMERIC,    -- ДРР Total (факт/выручка)
  drr_ad        NUMERIC,    -- ДРР рекламный
  ctr           NUMERIC,    -- CTR (клики/показы)
  cr_cart       NUMERIC,    -- CR в корзину
  cr_order      NUMERIC,    -- CR в заказ
  cpm           NUMERIC,    -- CPM, ₽
  cpc           NUMERIC,    -- CPC, ₽
  ad_order_share NUMERIC,   -- Доля рекламных заказов
  spp           NUMERIC,    -- СПП %
  -- Планы (только для первого дня загрузки)
  spend_plan    NUMERIC,    -- Затраты план, ₽
  drr_plan      NUMERIC,    -- ДРР план
  -- Служебное
  upload_id     UUID REFERENCES uploads(id),
  UNIQUE (sku_ms, metric_date)
);
CREATE INDEX idx_fsd_date ON fact_sku_daily(metric_date);
CREATE INDEX idx_fsd_sku_date ON fact_sku_daily(sku_ms, metric_date);
```

### 3. fact_sku_snapshot — снимок SKU на дату загрузки

**Источник:** Отчёт по SKU (1 снимок на загрузку = последний день файла)  
**Ключ:** (sku_ms, upload_id) — один снимок на загрузку

```sql
CREATE TABLE fact_sku_snapshot (
  id             BIGSERIAL,
  sku_ms         TEXT NOT NULL REFERENCES dim_sku(sku_ms),
  upload_id      UUID NOT NULL REFERENCES uploads(id),
  snap_date      DATE,
  -- Остатки (из Отчёта)
  fbo_wb         INTEGER,   -- Остаток ФБО ВБ
  fbs_pushkino   INTEGER,   -- Остаток FBS Пушкино
  fbs_smolensk   INTEGER,   -- Остаток FBS Смоленск
  kits_stock     INTEGER,   -- Остаток комплектов
  stock_days     INTEGER,   -- Остаток дни (с учётом склада)
  days_to_arrival INTEGER,  -- Дней до прихода
  ots_reserve_days INTEGER, -- Запас дней до OOS
  -- Экономика
  margin_rub     NUMERIC,   -- Маржа операционная, ₽ (Колонка X)
  chmd_5d        NUMERIC,   -- ЧМД за 5 дней, ₽ (Колонка Z)
  price          NUMERIC,   -- Текущая цена
  -- Планы
  spend_plan     NUMERIC,
  drr_plan       NUMERIC,
  -- Поставки
  supply_date    DATE,
  supply_qty     INTEGER,
  -- Характеристики
  shelf_date     DATE,
  manager        TEXT,
  novelty_status TEXT,
  UNIQUE (sku_ms, upload_id)
);
CREATE INDEX idx_fss_upload ON fact_sku_snapshot(upload_id);
CREATE INDEX idx_fss_sku ON fact_sku_snapshot(sku_ms);
```

### 4. fact_stock_snapshot — снимок остатков из Таблицы остатков

**Источник:** Таблица остатков  
**Ключ:** (sku_wb, upload_id)

```sql
CREATE TABLE fact_stock_snapshot (
  id           BIGSERIAL,
  sku_wb       BIGINT NOT NULL,      -- WB артикул (главный ключ)
  sku_ms       TEXT,                 -- sku_ms из dim_sku (заполняется автоматически)
  upload_id    UUID NOT NULL REFERENCES uploads(id),
  snap_date    DATE,
  fbo_wb       INTEGER,
  fbs_pushkino INTEGER,
  fbs_smolensk INTEGER,
  total_stock  INTEGER,              -- ADC (Всего = ФБО + ФБС + комплекты)
  price        NUMERIC,
  margin_pct   NUMERIC,
  supply_qty   INTEGER,
  supply_date  DATE,
  UNIQUE (sku_wb, upload_id)
);
CREATE INDEX idx_fssnap_upload ON fact_stock_snapshot(upload_id);
CREATE INDEX idx_fssnap_wb ON fact_stock_snapshot(sku_wb);
```

ALTER TABLE dim_sku ADD PRIMARY KEY (sku_wb);

## Порядок загрузки после пересоздания

**ОБЯЗАТЕЛЬНЫЙ порядок:**

1. **Свод** (catalog) — заполняет dim_sku. БЕЗ этого ничего не работает.
2. **Таблица остатков** (stock) — fact_stock_snapshot + fact_stock_daily + fact_price_changes
3. **Отчёт по SKU** (sku_report) — fact_sku_daily + fact_sku_snapshot. Парсер берёт col 0 = sku_ms, ищет в dim_sku → получает sku_wb для обогащения.
4. **ABC анализ** (abc) — fact_abc (вторичное)
5. **Потребность Китай** (china) — fact_china_supply (вторичное)

## Что нужно исправить в парсере Свода (parseCatalog.ts)

Текущий парсер ищет `'артикул мс'` и записывает в `sku_ms`. Ты говоришь что убираешь «Артикул склада» и оставляешь только «Артикул МС» — это правильно.

**Свод должен содержать:**
| Колонка Excel | Что ищет парсер | Куда записывает |
|---|---|---|
| Артикул WB | `'артикул wb'` | `dim_sku.sku_wb` (PRIMARY KEY) |
| Артикул МС | `'артикул мс'`, `'артикул mc'` | `dim_sku.sku_ms` (UNIQUE) |
| Номенклатура | `'номенклатура'`, `'название'` | `dim_sku.name` |
| Бренд | `'бренд'` | `dim_sku.brand` |
| Предмет WB | `'предмет wb'` | `dim_sku.subject_wb` |
| Категория WB | `'категория wb'` | `dim_sku.category_wb` |
| Поставщик | `'поставщик'` | `dim_sku.supplier` |
| Январь..Декабрь | `'январь'`..`'декабрь'` | `dim_sku.month_jan`..`month_dec` |

**Что убираем:** Артикул склада (= sku_ms, дублирование), Артикул Китай (не используется)

## Почему сейчас подтянулись только частично

`fact_sku_snapshot` содержит 49,585 строк — это правильно (17 загрузок × ~2763).
НО в API-роутах используется только ПОСЛЕДНИЙ snapshot (last upload_id по типу 'sku_report').
Поэтому маржа и ЧМД видны только для последней загрузки (01.04–05.04).

Это **правильное поведение** — снапшот = актуальный срез, не исторический.
Исторические метрики = fact_sku_daily (выручка, ДРР, CTR по дням).
