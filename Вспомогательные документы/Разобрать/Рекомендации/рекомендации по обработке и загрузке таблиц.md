В Excel может быть:

✅ Решение: нормализация колонок
const COLUMN_MAP = {
  "артикул": "sku_supplier",
  "артикул поставщика": "sku_supplier",
  "sku": "sku",
  "дата": "date",
  "расход": "ad_spend"
};
Преобразование:
function normalizeRow(row) {
  const normalized = {};

  for (const key in row) {
    const cleanKey = key.trim().toLowerCase();

    if (COLUMN_MAP[cleanKey]) {
      normalized[COLUMN_MAP[cleanKey]] = row[key];
    }
  }

  return normalized;
}
❗ Это решает:
разный язык колонок
лишние пробелы
хаос в Excel
4. Как НЕ тянуть лишние колонки
❌ Ошибка:
insert(row) // всё подряд
✅ Правильно:
Ты берёшь только whitelist полей
const REQUIRED_FIELDS = [
  "sku_supplier",
  "date",
  "ad_spend",
  "clicks"
];
function filterRow(row) {
  const result = {};

  for (const field of REQUIRED_FIELDS) {
    result[field] = row[field] ?? null;
  }

  return result;
}

Как считается
1. Джойн всех фактов
SELECT
    s.sku_id,
    s.date,
    s.revenue,
    s.orders,
    a.ad_spend,
    a.clicks,
    st.stock_qty
FROM fact_sales s
LEFT JOIN fact_ads a
    ON s.sku_id = a.sku_id AND s.date = a.date
LEFT JOIN fact_stock st
    ON s.sku_id = st.sku_id AND s.date = st.date
2. Производные метрики
profit = revenue - ad_spend
margin = profit / revenue
cpo = ad_spend / orders
ctr = clicks / impressions (если есть)
3. Тренды
AVG(revenue) OVER (PARTITION BY sku_id ORDER BY date ROWS BETWEEN 6 PRECEDING AND CURRENT ROW)
revenue_diff_pct =
(revenue_3d_avg - revenue_7d_avg) / revenue_7d_avg
4. Оборачиваемость
days_of_stock = stock_qty / (qty_7d_avg)

3) Auto CPO (ключевая штука)
Ты считаешь максимально допустимый CPO.
3.1. Формула
target_cpo = margin_per_order * k
где:
margin_per_order = (price - cost - logistics)
3.2. Коэффициент k
если рост → k = 1.2 (можно агрессивнее)
если падение → k = 0.8
если нестабильно → k = 1.0
SQL:
target_cpo =
CASE
  WHEN revenue_diff_pct > 0.2 THEN margin * 1.2
  WHEN revenue_diff_pct < -0.2 THEN margin * 0.8
  ELSE margin
END
3.3. Учет прогноза
Усиливаем:
IF pred_revenue растет → увеличиваем target_cpo
IF падает → уменьшаем

Сейчас у меня очень хорошо считаются данные для заказа, учитывается сезонность, коэффициент каждого месяца, используются продажи прошлых периодов и т.д. А как сделать что то подобное, только с прогнозом продаж? В целом я вижу это так - использовать данные по заказу как прогноз продаж. В целом так оно и есть, как мне кажется и выводить информацию с прогнозом продаж и фактом. Так мы сможем видеть план, факт и прогноз по нашим расчетам. Оформить в виде графика и добавить минимальные какие то метрики
Ты мыслишь в правильном направлении — логика заказа = уже готовый прогноз, просто в “операционной форме”.
Но есть нюанс: заказ — это спрос + ограничения (stock, MOQ, lead time), а прогноз продаж — это чистый спрос.
👉 Поэтому правильно:
разделить “прогноз спроса” и “расчёт заказа”, но использовать одну базу логики.

1. Как трансформировать твою модель заказа → в прогноз
Сейчас у тебя (упрощённо):
заказ = прогноз спроса × коэффициенты сезонности × период поставки ± корректировки
👉 значит у тебя уже есть:
базовый спрос
сезонные коэффициенты
тренды
✅ Правильная декомпозиция
Раздели это на 2 слоя:

2. order_calc (поверх прогноза)

3. Как считать прогноз (на базе твоей логики)
Шаг 1. Базовый спрос
base_demand = avg(qty за последние 30 дней)
или лучше:
base_demand = weighted_avg(7d, 14d, 30d)
Шаг 2. Сезонность
У тебя уже есть коэффициенты:
seasonality_coef = month_coef
Шаг 3. Тренд
trend_coef =
CASE
  WHEN revenue растет THEN 1.1
  WHEN падает THEN 0.9
  ELSE 1
END
Шаг 4. Финальный прогноз
pred_qty =
base_demand * seasonality_coef * trend_coef
❗ Ключевое отличие от заказа
В заказе есть:
lead_time
остатки
страховой запас
👉 В прогнозе этого НЕ должно быть
4. Почему твоя идея “использовать заказ как прогноз” частично неверна
❌ Проблема:
Если ты берёшь заказ как прогноз:
он уже включает страховой запас
он зависит от текущих остатков
он искажается логистикой
✅ Правильно:
прогноз = база
заказ = функция(прогноз, stock, lead_time)

8. Минимальные метрики (не перегружай)
1. Accuracy (точность прогноза)
forecast_error =
ABS(fact - forecast) / fact
2. Bias (системная ошибка)
bias =
(forecast - fact) / fact
3. Выполнение плана
plan_vs_fact =
fact / plan
4. Отклонение от прогноза
forecast_vs_fact =
fact / forecast

6) Автоматическая калибровка (ключ к качеству)
6.1. Ошибка прогноза
error = (fact_qty - pred_qty) / NULLIF(fact_qty, 0)
6.2. Bias (смещение)
bias_7d = AVG(error) за 7 дней
6.3. Коррекция
corrected_pred =
pred_qty * (1 + bias_7d)
👉 если модель стабильно недооценивает → поднимем прогноз
👉 если переоценивает → уменьшим
